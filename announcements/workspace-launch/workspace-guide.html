MFL Workspace

The new umbrella home for every MFL repo — shared context, unified Claude Code setup, and the central knowledge system. This guide replaces mfl-knowledge-base.

Why this change

As MFL grows, our docs, projects, and tooling have spread across multiple repos and tools. mfl-knowledge-base was a single-purpose wiki; it no longer fits the way we actually work — multiple teams, multiple projects, and AI agents that need consistent context across all of them.

mfl-workspace replaces it. It's an umbrella repo that sits at ~/mfl-workspace/ and holds every domain-specific repo as a sibling folder. Claude Code automatically loads the shared workspace context on top of each repo's own instructions — so every agent across the company shares the same red lines, terminology, and conventions without anyone having to configure it.

Workspace structure

The workspace is a flat collection of independent Git repos, each owned by a team or function. Every repo carries its own CLAUDE.md; the workspace-level CLAUDE.md cascades down to all of them.

Getting started

Three commands and you're set up:

git clone https://github.com/Metaverse-Football-League/mfl-workspace.git ~/mfl-workspace
cd ~/mfl-workspace
./setup.sh

The setup script clones every sub-repo you have access to and silently skips the ones you don't. It also symlinks general-purpose skills (/commit, /project-create, /agent-browser, /email) so they're available globally — in any repo, on any project.

From there, working is simple:

cd mfl-marketing
claude

Claude Code auto-loads the workspace CLAUDE.md + the repo's own CLAUDE.md. No extra config.

How Claude Code uses the workspace

Two kinds of context are loaded automatically whenever you launch Claude inside a sub-repo:

  • Workspace-level (~/mfl-workspace/CLAUDE.md) — red lines (no "investment" framing, 18+ strict, no invented facts), critical terminology (club owner, player, $MFL, manager), and the repo catalog.
  • Repo-level (mfl-<domain>/CLAUDE.md) — domain-specific conventions, data sources, workflows, and the skills available for that team.

Cross-repo references in skills and docs use relative paths (e.g. ../mfl-wiki/knowledge-base/09-marketing/), which is why all repos must sit side-by-side.

Focus on mfl-wiki Core

mfl-wiki is the heart of our knowledge system. Everything we learn — from a user interview to a partnership deck to a tokenomics note — eventually lands here so that any team member (and any agent) can find it.

Three-layer structure

knowledge-base/

Curated, canonical pages organized by domain (e.g. 09-marketing/, numbered to keep ordering stable). This is the trusted reference.

sources/

Raw imported material — interview transcripts, articles, internal docs, decks. Unedited, timestamped, attributed.

memory/

Long-term agent memory — stable facts, preferences, and context agents can draw on across sessions.

How the wiki is fed

Feeding the wiki is a deliberate, three-step flow so nothing leaks into the KB unchecked:

  1. /source-import — pulls raw material into sources/. Any format: URLs, PDFs, transcripts, pasted text.
  2. /wiki-ingest — curates a source into the knowledge-base/: normalizes terminology, places it in the right domain, links related pages, drops anything off-brand or unverified.
  3. /wiki-update — evolves existing KB pages with new facts or corrections. Preserves history.

A fourth skill, /wiki-recap, gives you a digest of what changed recently — useful for weekly reviews or catching up after time off.

Red line: the wiki is the source of truth. If a claim isn't verifiable in knowledge-base/ or traceable to a file in sources/, agents must flag it as unverified rather than invent. Accuracy first.

Red lines (always enforced)

  • Never position MFL as investment — no "you can make money" or "guaranteed returns." MFL is a game.
  • 18+ restriction is strict — real-money economy requires it.
  • Don't overpromise features — only discuss what is officially announced.
  • Be careful with economic claims — verify against KB source files.
  • Never invent facts — accuracy first. If unsure, say so.
  • English only — all files, commit messages, and documentation must be in English.
